# Devctx Auto-Orchestration Design

## Status

Proposed design for phased implementation.

## Problem

`devctx` already has strong primitives:

- `smart_turn` for start/end orchestration
- `smart_context` / `smart_search` / `smart_read` for compressed retrieval
- `smart_summary` / SQLite for persisted continuity
- `task-runner`, `headless-wrapper`, and client init scripts for higher-level workflows

What is still missing is a unified automation layer. Today, orchestration logic is spread across:

- `tools/devctx/src/tools/smart-turn.js`
- `tools/devctx/src/workflow-tracker.js`
- `tools/devctx/src/task-runner.js`
- `tools/devctx/src/task-runner/policy.js`
- `tools/devctx/src/orchestration/headless-wrapper.js`
- client-specific config generated by `tools/devctx/scripts/init-clients.js`

This creates four problems:

1. Automaticity is inconsistent by client.
2. Policy is duplicated or partially duplicated.
3. Wrappers and hooks are harder to evolve safely.
4. Product claims about "near-automatic" usage are hard to measure rigorously.

## Design Goal

Increase automaticity without changing the MCP server's passive model.

The MCP must remain a passive tool server. Automatic behavior must come from an orchestration layer above it, driven by hooks, wrappers, or client launchers.

## Non-Goals

- Do not make the MCP intercept prompts directly.
- Do not persist after every raw message.
- Do not move business logic into client-specific adapters.
- Do not replace `smart_turn`; adapters must orchestrate it, not fork its semantics.

## Principles

### 1. Passive Core, Active Edge

The MCP remains passive. Automaticity lives at the edge:

- client hooks
- terminal wrapper
- assisted launcher
- task runner

### 2. One Policy, Many Adapters

Automaticity decisions must come from one shared policy engine. Adapters should translate client events into shared orchestration events and execute the result.

### 3. Event-Driven Persistence

Preserve the current event-driven model:

- session start
- task handoff
- meaningful progress
- checkpoint-worthy milestone
- task stop / graceful end

Do not save on every prompt.

### 4. Measure Net Benefit

Automaticity is only successful if:

- continuity improves
- skipped workflows decrease
- checkpoint quality remains high
- overhead stays below the value created

## Target Architecture

```text
Client / Wrapper / Hook
        |
        v
Adapter (client-specific)
        |
        v
Base Orchestrator
        |
        +--> Event Policy
        +--> smart_turn
        +--> smart_context / smart_search / smart_read
        +--> smart_summary
        +--> workflow-tracker / metrics
```

## Responsibilities

### `base-orchestrator.js`

Owns the shared orchestration lifecycle.

Responsibilities:

- run `smart_turn(start)`
- choose preflight action
- provide context package for the wrapped execution
- evaluate whether an end checkpoint should run
- run `smart_turn(end)`
- record orchestration metrics

Must not:

- know anything about Claude/Cursor/Codex specifics
- parse client config files
- encode client event names directly

### `auto-orchestrator.js`

Thin facade over the base orchestrator for "managed mode".

Responsibilities:

- expose a stable entrypoint to wrappers and adapters
- accept normalized orchestration events
- apply automaticity level (`manual`, `low`, `medium`, `high`)

### `policy/event-policy.js`

Pure decision module. No I/O.

Responsibilities:

- determine `autoStart`
- determine `autoRefresh`
- determine `autoCheckpoint`
- expose reasons for `run`, `skip`, or `defer`

Example shape:

```js
export const ORCHESTRATION_POLICY = {
  autoStart: {
    runWhen: ['session_start', 'task_prompt', 'resume_prompt'],
    skipWhen: ['trivial_prompt', 'already_aligned_followup'],
  },
  autoRefresh: {
    runWhen: ['stale_index', 'task_switch', 'ambiguous_resume', 'context_mismatch'],
    skipWhen: ['aligned_continuation'],
  },
  autoCheckpoint: {
    runWhen: ['milestone', 'task_complete', 'error_recovered', 'stop_hook'],
    skipWhen: ['read_only_exploration', 'weak_single_file_touch'],
  },
};
```

### `policy/workflow-policy.js`

Extract workflow classification logic currently spread across:

- `workflow-tracker.js`
- `task-runner/policy.js`

Responsibilities:

- classify workflow type
- choose preflight tool family
- define minimal context package per workflow

### `policy/checkpoint-policy.js`

Extract "should persist / why / with what event" semantics from `smart-turn` and `smart-summary` orchestration call sites.

Responsibilities:

- map execution outcomes to end events
- decide between `milestone`, `checkpoint`, `task_complete`, `error_recovered`, `stop`
- standardize skip reasons

### `adapters/*.js`

One adapter per client or execution surface.

Responsibilities:

- ingest native client event
- normalize it into shared orchestration event format
- call `auto-orchestrator`
- map returned guidance back into the client surface

Adapters must stay thin.

## Core Contracts

### Normalized Client Event

```js
{
  source: 'claude' | 'cursor' | 'codex' | 'qwen' | 'task_runner' | 'headless',
  kind: 'session_start' | 'user_prompt' | 'post_tool_use' | 'stop' | 'resume',
  prompt: string | null,
  sessionId: string | null,
  metadata: {
    toolName?: string,
    touchedFiles?: string[],
    exitCode?: number,
    clientSessionId?: string,
  },
}
```

### Preflight Decision

```js
{
  action: 'none' | 'smart_context' | 'smart_search',
  reason: string,
  params: object,
}
```

### Orchestrator Result

```js
{
  start: object | null,
  preflight: object | null,
  guidance: {
    continuityState: string,
    nextTools: string[],
    topFiles: string[],
    instructions: string[],
  },
  checkpoint: {
    attempted: boolean,
    persisted: boolean,
    event: string | null,
    reason: string,
  } | null,
  metrics: {
    overheadTokens: number,
    preflightUsed: boolean,
    checkpointUsed: boolean,
  },
}
```

## Integration Points with Existing Code

### Reuse Without Forking

- `smart_turn` remains source of truth for start/end orchestration.
- `smart_context` and `smart_search` remain source of truth for preflight retrieval.
- `smart_summary` remains the persistence substrate.
- `workflow-tracker` remains the analytics substrate, but some policy logic should move out.

### Existing Files to Refactor

- `tools/devctx/src/task-runner.js`
  Should consume `base-orchestrator` instead of embedding orchestration logic.

- `tools/devctx/src/orchestration/headless-wrapper.js`
  Should become a wrapper over `base-orchestrator` rather than a parallel implementation.

- `tools/devctx/src/workflow-tracker.js`
  Keep tracking, reduce policy ownership.

- `tools/devctx/src/task-runner/policy.js`
  Keep command/workflow definitions, move generic event policy out.

## Execution Flows

### 1. Claude Desktop with Hooks

Best automaticity target.

Flow:

1. Hook emits `session_start` event.
2. `claude-adapter` normalizes event.
3. `auto-orchestrator` runs `smart_turn(start)`.
4. Policy decides preflight.
5. Guidance is injected into prompt or wrapper context.
6. `post_tool_use` events accumulate meaningful progress.
7. On `stop` or qualifying `post_tool_use`, orchestrator runs `smart_turn(end)`.

### 2. Cursor Assisted Launcher

Flow:

1. `cursor-devctx` launcher wraps the task.
2. `cursor-adapter` sends normalized `user_prompt` event.
3. `auto-orchestrator` runs start and preflight.
4. Wrapped agent receives prompt plus operational context.
5. End checkpoint runs when launcher completes.

### 3. Codex / Qwen Managed Wrapper

Flow:

1. User runs `codex-devctx run --prompt "..."`
2. Wrapper emits normalized `user_prompt`.
3. `auto-orchestrator` executes start + preflight.
4. Wrapped command receives context package.
5. Wrapper collects output metadata.
6. Orchestrator decides whether to checkpoint.

## Automaticity Levels

```json
{
  "automaticity": "manual | low | medium | high"
}
```

Expected meaning:

- `manual`
  No auto-start, no auto-checkpoint.

- `low`
  Auto-start on obvious non-trivial prompts only.

- `medium`
  Auto-start + selective preflight + selective checkpoint.

- `high`
  Hook/wrapper-driven start, preflight, and end for most non-trivial turns.

This must be client-aware. `high` in Claude is realistic. `high` in Codex/Qwen still means wrapper-managed, not invisible background automation.

## Metrics

Add automaticity metrics early.

### New metrics to record

- `autoStartRatePct`
- `autoRefreshRatePct`
- `autoCheckpointRatePct`
- `checkpointPrecisionPct`
- `resumeAlignmentRatePct`
- `orchestrationOverheadTokens`
- `netAutomaticityBenefitTokens`
- `managedModeUsageCount`

### Success criteria

- Auto-start should increase continuity without large overhead.
- Checkpoint precision must remain high.
- Managed mode must outperform manual mode on non-trivial tasks.

## Phased Implementation Plan

### Phase 1: Consolidate Policy and Shared Orchestration

Goal: Remove duplicated logic.

Deliverables:

- `policy/event-policy.js`
- `policy/workflow-policy.js`
- `policy/checkpoint-policy.js`
- `base-orchestrator.js`
- automaticity metrics plumbing

Refactors:

- `task-runner.js` uses `base-orchestrator`
- `headless-wrapper.js` uses `base-orchestrator`

### Phase 2: First-Class Client Adapters

Goal: Move from wrappers with embedded assumptions to client-aware adapters.

Deliverables:

- `claude-adapter.js`
- `cursor-adapter.js`
- `codex-adapter.js`
- `qwen-adapter.js`

Priority order:

1. Claude
2. Cursor
3. Codex
4. Qwen

### Phase 3: Managed Mode

Goal: Stable product experience with configurable automaticity.

Deliverables:

- `auto-orchestrator.js`
- managed mode config
- benchmark scenarios for automaticity
- docs and compatibility matrix updates

## Risks and Mitigations

### Risk: Adapter logic duplicates orchestration logic

Mitigation:

- keep adapters thin
- centralize all decisions in `base-orchestrator` + policy modules

### Risk: Automation overhead exceeds token savings

Mitigation:

- record overhead at every managed step
- benchmark `manual` vs `managed`

### Risk: Passive MCP model gets blurred

Mitigation:

- keep all automaticity outside the MCP server
- do not add any autonomous loop into `server.js`

### Risk: Managed mode becomes opaque

Mitigation:

- every orchestrator decision should emit a reason
- expose those reasons through metrics and logs

## First Implementation Slice

The first slice should be deliberately small:

1. Create `policy/event-policy.js`
2. Create `base-orchestrator.js`
3. Refactor `task-runner.js` to consume it
4. Refactor `headless-wrapper.js` to consume it
5. Add automaticity metrics

This gives immediate value without waiting for client adapters.

## Open Questions

- Should `workflow-tracker` remain only analytics, or still own some workflow detection helpers?
- Should automaticity config live in `.devctx/config.json`, per client config, or both?
- How much client-specific prompting should adapters own versus `client-contract.js`?
- Should managed mode expose dry-run reasoning for debugging orchestration decisions?

## Decision

Use a hexagonal orchestration layer:

- passive MCP core unchanged
- shared base orchestrator as application layer
- shared policy modules as domain decision layer
- thin client adapters as interface adapters

That is the highest-automaticity path that still respects the current product architecture.
